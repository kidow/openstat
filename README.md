# OPENSTAT 계획 문서

## 1. 프로젝트 개요

OPENSTAT은 대한민국 정부 데이터를 기반으로 다양한 통계 차트를 한 화면에서 탐색할 수 있는 대시보드입니다.
초기 MVP는 홈페이지만 구축하며, 하나의 페이지에서 여러 차트를 그리드 형태로 보여주는 것이 목표입니다.

이 프로젝트의 핵심은 단순한 차트 전시가 아니라, 향후 매우 많은 차트가 추가되더라도 안정적으로 운영할 수 있는 캐싱 중심 구조를 먼저 설계하는 것입니다.
정적 생성과 재검증은 그 전략의 일부이며, 필요에 따라 태그 기반 무효화, 캐시 프로필 분리, 온디맨드 갱신 등 더 적극적인 전략을 함께 사용합니다.

핵심 방향은 다음과 같습니다.

- Next.js 기반의 캐싱 우선 대시보드
- TypeScript를 사용한 타입 안정성 확보
- Tailwind CSS + shadcn/ui 기반의 일관된 UI 구성
- shadcn/ui 차트 프리미티브가 사용하는 Recharts 채택
- 여러 차트를 그리드 레이아웃으로 배치
- 차트별로 독립적인 데이터 캐시 정책 적용
- 라이트/다크 테마를 shadcn/ui 기본 스타일 체계에 맞춰 지원

## 2. MVP 목표

MVP의 목표는 "홈 한 페이지에서 여러 차트를 안정적으로 보여주는 구조를 완성하고, 이후 차트 수가 크게 늘어나도 유지 가능한 캐싱 전략의 기반을 만드는 것"입니다.

포함 범위:

- 홈 페이지 1개 구현
- 상단 헤더 구현
- 그리드 기반 차트 카드 목록 구현
- 각 카드에 차트 1개씩 배치
- shadcn/ui 스타일 기반 차트 카드 시스템 구축
- Recharts 기반 차트 렌더링 구조 정착
- 차트별 독립 캐시 정책 설계
- 정적 생성, 시간 기반 재검증, 태그 기반 무효화 등 확장 가능한 데이터 계층 설계
- 라이트/다크 테마 지원

제외 범위:

- 로그인/권한 관리
- 관리자 페이지
- 차트 상세 페이지
- 검색, 필터, 정렬 고도화
- 실시간 스트리밍 데이터
- 사용자별 개인화 기능

## 3. 기술 스택

- Framework: Next.js
- Language: TypeScript
- Styling: Tailwind CSS
- UI Components: shadcn/ui
- Chart Library: Recharts
- Theme System: shadcn/ui 기본 테마 패턴 사용

차트 라이브러리는 Recharts로 확정합니다.
이유는 shadcn/ui의 차트 예제가 Recharts를 기준으로 구성되어 있고, 카드 기반 대시보드 UI와 결합하기 쉽기 때문입니다.

초기 범위에서 사용하는 차트 유형은 다음과 같습니다.

- Area Chart
- Bar Chart
- Line Chart
- Pie Chart
- Radar Chart
- Radial Chart

즉, 홈 MVP는 단순히 한두 종류의 차트만 올리는 페이지가 아니라, shadcn/ui에서 일반적으로 다루는 주요 차트 유형 전부를 수용할 수 있는 공통 구조를 목표로 합니다.

## 4. 제품 요구사항 정리

### 화면 구조

- 전체 화면 사용
- 상단 헤더 높이 44px
- 헤더 좌측에 텍스트 로고 `OPENSTAT`
- 헤더에는 불필요한 탐색 요소 없이 로고만 배치
- 본문은 그리드 레이아웃
- 각 그리드 아이템은 차트 카드 1개
- 차트 카드 UI는 shadcn/ui 스타일 사용
- 홈 한 페이지에서 여러 차트를 동시에 보여줌

### 데이터 및 운영 방향

- 데이터 소스는 대한민국 정부 데이터 중심
- 대표 도메인은 부동산, 주식, 경제 등 공공 데이터 계열
- 모든 차트는 서버 중심 데이터 준비를 우선 고려
- 차트마다 데이터 변경 주기와 캐싱 정책이 다를 수 있음
- 차트 수가 많아져도 전체 페이지 비용이 급격히 늘지 않도록 설계해야 함

### 테마

- 라이트 테마 지원
- 다크 테마 지원
- shadcn/ui에서 사용하는 기본 스타일 감각을 유지
- 별도 커스텀 디자인 시스템을 먼저 만들기보다 shadcn/ui 기준을 따름

## 5. 최우선 설계 원칙

이 프로젝트의 최우선 목표는 "차트별 캐싱 정책을 적극적으로 차용할 수 있는 구조"입니다.
홈 화면이 먼저이지만, 구조는 처음부터 다수의 차트를 감당할 수 있어야 합니다.

따라서 MVP에서도 다음 원칙을 적용합니다.

- 페이지 전체보다 차트 단위를 우선해서 설계
- 데이터 fetch와 캐시 범위를 차트별로 분리
- 동일 데이터 계열은 태그로 묶고, 필요 시 부분 무효화 가능하게 구성
- 시간 기반 재검증만 쓰지 않고 온디맨드 무효화도 고려
- 차트 UI와 데이터 접근 계층을 분리
- 차트 추가 시 기존 차트 캐시 정책에 영향이 최소화되도록 구성

## 6. 권장 아키텍처

MVP에서는 홈 페이지를 서버 컴포넌트 중심으로 구성하고, 차트별 데이터 조회와 렌더링 책임을 카드 단위로 분리하는 방향이 적절합니다.
차트 수가 많아질 것을 고려하면, 페이지 하나에서 모든 데이터를 한꺼번에 처리하는 구조는 피하는 것이 좋습니다.

권장 구조:

- `app/page.tsx`
  - 홈 페이지 진입점
  - 헤더 + 메인 그리드 구성
- `components/layout/site-header.tsx`
  - 44px 헤더
  - 텍스트 로고 표시
- `components/charts/chart-grid.tsx`
  - 전체 차트 카드 배치
- `components/charts/chart-card.tsx`
  - shadcn/ui `Card` 기반 공통 차트 카드
- `components/charts/*`
  - 개별 차트 프레젠테이션 컴포넌트
- `lib/charts/registry.ts`
  - 차트 메타데이터 목록
- `lib/charts/data/*`
  - 차트별 데이터 로더
- `lib/charts/cache/*`
  - 캐시 정책, 태그, 프로필 정의
- `lib/types/*`
  - 차트 데이터 타입 정의

권장 분리 원칙:

- 차트 설정 메타데이터와 렌더링 컴포넌트 분리
- 데이터 로더와 시각화 컴포넌트 분리
- 캐시 태그와 재검증 규칙을 화면 코드에서 분리
- 공통 카드 UI와 개별 차트 구현 분리

## 7. 캐싱 및 재검증 전략

이 프로젝트는 `revalidate`만으로 설명하기 어렵고, Next.js의 여러 캐시 계층을 조합해서 보는 것이 맞습니다.
공식 문서 기준으로 현재는 `fetch` 캐시, 전체 라우트 캐시, 클라이언트 라우터 캐시, `use cache`, `cacheTag`, `revalidateTag`, `revalidatePath`, `cacheLife` 등을 함께 설계 대상으로 보는 것이 적절합니다.
또한 `unstable_cache`는 레거시 성격이므로 신규 구조의 중심에는 두지 않는 편이 좋습니다.

### 7.1 전략 우선순위

우선순위는 다음과 같습니다.

1. 차트별 독립 데이터 캐시
2. 태그 기반 캐시 무효화
3. 시간 기반 재검증
4. 정적 페이지 및 전체 라우트 캐시 활용
5. 필요 시 경로 단위 무효화

### 7.2 적용 원칙

- 홈 페이지 자체는 정적 렌더링 친화적으로 유지
- 각 차트 데이터는 별도 데이터 함수로 분리
- 각 차트는 자체 캐시 정책을 가짐
- 동일한 정부 데이터 계열은 공통 태그를 공유 가능
- 차트별 세부 태그를 추가해 부분 갱신 가능
- 변경 빈도가 낮은 차트는 더 긴 캐시 수명 적용
- 갱신 이벤트가 명확한 데이터는 태그 기반 온디맨드 무효화 우선 고려

### 7.3 권장 캐시 도구

Next.js 공식 문서 기준으로 다음 도구를 우선 검토합니다.

- `fetch(..., { cache: 'force-cache' })`
  - 요청 자체를 캐싱할 때 사용
- `fetch(..., { next: { revalidate: number, tags: [...] } })`
  - 시간 기반 재검증과 태그 지정이 필요한 경우 사용
- `use cache`
  - fetch 외의 계산/조합 로직까지 캐시 범위에 포함시키고 싶을 때 사용
- `cacheLife()`
  - 차트 특성에 따라 캐시 프로필을 세분화할 때 사용
- `cacheTag()`
  - 차트 또는 데이터 계열에 태그 부여
- `revalidateTag()`
  - 특정 차트군 또는 데이터군만 선택적으로 무효화
- `revalidatePath()`
  - 경로 단위 무효화 필요 시 사용

### 7.4 실무 적용 예시

예시 1. 자주 바뀌지 않는 월간 부동산 통계

- 정적 렌더링 우선
- 긴 캐시 수명 사용
- 월간 데이터 태그 부여
- 데이터 갱신 시에만 태그 무효화

예시 2. 상대적으로 자주 갱신되는 경제 지표

- 요청 캐시 사용
- 짧은 시간 기반 재검증 사용
- 도메인 태그와 차트 태그를 함께 부여

예시 3. 여러 차트가 같은 원천 데이터를 공유하는 경우

- 원천 데이터 로더를 공통 캐시 범위로 분리
- 파생 차트는 같은 태그 체계를 공유
- 데이터 갱신 시 관련 차트 전체만 재검증

### 7.5 문서상 기본 방침

문서와 코드에서 기본적으로 다음을 전제합니다.

- 차트별 캐시 정책은 메타데이터로 선언 가능해야 함
- `revalidate`는 여러 전략 중 하나일 뿐이며 기본 전략 전체를 대표하지 않음
- 캐시 정책은 차트 UI 코드 안에 하드코딩하지 않음
- 새 차트를 추가할 때 캐시 태그, 수명, 무효화 전략을 함께 정의할 수 있어야 함

## 8. 데이터 소스 전략

초기 데이터는 대한민국 정부 및 공공 데이터 계열을 기준으로 설계합니다.
도메인은 크게 다음 범주를 상정합니다.

- 부동산
- 주식 및 시장 지표
- 경제 일반
- 인구/고용 등 행정 통계
- 기타 공공 데이터 API

데이터 소스 특성상 다음을 염두에 둡니다.

- 데이터 제공 주기가 서로 다를 수 있음
- 원본 API 응답 형태가 일관되지 않을 수 있음
- 차트용 정규화 레이어가 필요함
- 원천 데이터 캐시와 시각화 데이터 캐시를 분리할 여지가 있음

권장 계층:

- Source Layer: 외부 공공 데이터 수집
- Normalize Layer: 차트 친화적 구조로 정규화
- Cache Layer: 차트별 또는 도메인별 캐시 적용
- Presentation Layer: Recharts로 시각화

## 9. UI/UX 방향

### 헤더

- 높이 44px 고정
- 좌측 정렬
- 로고는 텍스트 `OPENSTAT`
- 불필요한 네비게이션 없이 단순한 상단 바 유지

### 메인 콘텐츠

- 화면 전체 폭 사용
- 적절한 외곽 여백 적용
- 카드형 그리드 배치
- 데스크톱에서는 다열 그리드
- 태블릿/모바일에서는 1열 또는 2열로 축소

### 차트 카드

각 카드에는 최소한 다음 정보가 포함되는 것이 좋습니다.

- 차트 제목
- 짧은 설명 또는 기준 시점
- 차트 본문
- 마지막 갱신 시점 또는 데이터 기준 정보
- 데이터 출처 라벨
- 캐시 또는 갱신 기준을 설명하는 보조 정보

shadcn/ui 활용 방향:

- `Card`
- `CardHeader`
- `CardTitle`
- `CardDescription`
- `CardContent`

### 테마

- shadcn/ui의 라이트/다크 토큰 체계를 그대로 사용
- 차트 색상도 테마 토큰에 맞춰 조정
- 다크 모드에서 축, 라벨, 그리드 색 대비를 별도 점검

## 10. 반응형 레이아웃 제안

초기 MVP에서는 아래와 같은 그리드 전략을 권장합니다.

- 모바일: 1열
- 태블릿: 2열
- 데스크톱: 3열
- 대형 화면: 4열

Tailwind 예시 방향:

- `grid-cols-1`
- `md:grid-cols-2`
- `xl:grid-cols-3`
- `2xl:grid-cols-4`

차트 높이는 카드별로 일정하게 관리하는 것이 MVP 단계에서 안정적입니다.

예시:

- 기본 카드 높이 고정
- 긴 설명은 최소화
- 차트 영역에 충분한 세로 공간 확보
- Pie/Radar/Radial 계열도 카드 높이 규칙을 공유

## 11. 데이터 모델 제안

차트가 계속 늘어날 수 있으므로, 차트 메타데이터를 선언형으로 관리하는 방식이 유리합니다.

권장 필드 예시:

- `id`
- `title`
- `description`
- `chartType`
- `domain`
- `sourceKey`
- `cacheStrategy`
- `cacheTags`
- `revalidate`
- `themeVariant`
- `component`

권장 예시 개념:

- `chartType`: area, bar, line, pie, radar, radial
- `domain`: real-estate, stocks, economy
- `cacheStrategy`: static, timed, tagged, hybrid
- `cacheTags`: domain 단위 + chart 단위 태그

이런 형태로 관리하면 다음이 쉬워집니다.

- 홈 페이지에서 차트 목록 순회 렌더링
- 캐시 정책 일원화 관리
- 차트 추가/순서 변경 간소화
- 특정 데이터군만 선택적 무효화

## 12. 구현 단계 제안

### 1단계. 프로젝트 기반 구성

- Next.js + TypeScript + Tailwind CSS 초기화

## 14. MVP 완료 기준

아래 조건을 만족하면 MVP 1차 완료로 봅니다.

- 홈 페이지가 정상 렌더링됨
- 헤더 높이가 44px로 적용됨
- 좌측에 `OPENSTAT` 텍스트 로고가 보임
- 본문이 그리드 형태로 구성됨
- 각 카드에 차트 1개씩 표시됨
- 카드 UI가 shadcn/ui 스타일 기준으로 정리됨
- Recharts 기반 차트 렌더링 구조가 마련됨
- Area, Bar, Line, Pie, Radar, Radial 차트 수용 구조가 정의됨
- 차트별 데이터 로딩 구조가 분리되어 있음
- 차트마다 다른 캐시 전략을 적용할 수 있음
- 라이트/다크 테마가 모두 동작함
- 모바일과 데스크톱에서 레이아웃이 무너지지 않음

## 16. 현재 시점의 결정 필요 항목

현재 계획만으로도 구현은 시작할 수 있지만, 아래 항목이 정해지면 더 구체적인 설계가 가능합니다.

- 초기 홈에 실제로 노출할 차트 개수
- 먼저 연결할 정부 데이터 API 목록
- 차트별 우선순위 도메인
- 태그 무효화를 트리거할 운영 방식
- 홈에서 보여줄 데이터 출처/기준 시점 표기 방식

## 17. 제안

현 시점에서는 다음 순서로 진행하는 것이 가장 효율적입니다.

1. Recharts + shadcn/ui 조합을 기준으로 홈 레이아웃과 차트 카드 규격부터 고정
2. 차트 메타데이터에 캐시 정책 필드를 먼저 설계
3. 정부 데이터 도메인을 부동산, 주식, 경제 3개 축으로 나누고 태그 체계 설계
4. 홈 MVP에는 대표 차트를 우선 배치하고, 나머지는 같은 구조로 확장
5. 시간 기반 재검증만 쓰지 말고 태그 기반 무효화까지 처음부터 수용 가능한 구조로 구현

## 18. 참고 메모

Next.js 공식 문서 기준으로 캐시/재검증 설계 시 참고할 핵심 개념은 다음과 같습니다.

- `fetch` 캐시와 `next.revalidate` / `next.tags`
- `use cache`, `cacheLife`, `cacheTag`
- `revalidateTag`, `revalidatePath`
- Full Route Cache
- Client Router Cache
- `unstable_cache`는 레거시로 간주하고 신규 설계의 중심에는 두지 않음
